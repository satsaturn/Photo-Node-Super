Lens Corrector — extension spec

The Lens module + the Lens Correction built-in edit. Profile files tagged by (focal length, aperture), linear interpolation, EXIF auto-detect, calibration workflow, lens library, JSON packs. Not in v1; the Lens tab is a disabled placeholder and the Lens Correction built-in slot is identity in v1.

Depends on v1 §6 (pipeline buckets — lens correction is bucket 0), §5 (layer header controls — the panel ports the enable/bypass pattern), §13 (catalog stores edit list + lens pack storage location), §14 (software stack).

1. Lens-correction pipeline integration

The lens-correction panel in Edit runs as the very first stage of the pipeline (order 0 in v1 §6), before crop, any built-in sheet, or any custom sheet. Rationale: geometric distortion changes the image geometry, so anything downstream (cropping, transforms, masks, blur kernels) needs the corrected coordinates. Vignetting and CA are pixel-value corrections, but keeping them in the same first stage means the corrector is a single, atomic module the user toggles. The panel references a lens profile file (copy or link, §4.6); a linked panel re-interpolates whenever the file is edited in the Lens section (§7).

The Lens Correction stage is "phase 0". Crop follows at stage 1 (see spec-crop). The four custom-sheet phases in v1 §6 are unchanged — they describe the custom pipeline, and both lens correction and crop are upstream of all of them. See Hole #31 for how this affects existing copy/link semantics on the first custom phase.

Bypass

The panel's Enable toggle (§4.2) bypasses the entire corrector: the photo enters the rest of the pipeline uncorrected. Identical semantics to a layer's enable toggle in the main editor (v1 §5).

Edge cases

  • No lens matched (EXIF unknown, Auto from EXIF off) → corrector shows an empty state and outputs the image unmodified. User can still pick a lens manually or create one.
  • Lens has zero profile points → corrector shows an empty state and outputs the image unmodified. A "+ New" CTA switches to the Lens section to start calibrating.
  • One profile point → that point's values are used directly (no interpolation).
  • Two or more points → linear interpolation per §5.

2. Lens Corrector — core concept & terminology

The Lens Corrector is the subsystem behind the Lens header module and the Lens Correction built-in edit in the Edit module. Per-lens profile files are tagged by (focal length, aperture), with linear interpolation between profile points. The lens-correction math (distortion, vignetting, CA, colour vignette) runs on the same CPU stack as the rest of the app (v1 §14).

Raw editors typically apply one static lens-correction profile per lens. But a lens's distortion and vignetting change with focal length and aperture, so a single profile over-corrects one image and under-corrects another shot on the same lens. The Lens Corrector fixes this by storing multiple profile points per lens — each tagged with a (focal length, aperture) pair — and linearly interpolating between the nearest points for every photo. The more points the user adds, the more accurate the result becomes. The system auto-detects the lens and shooting parameters from EXIF and applies the interpolated correction as the first step in the pipeline (§1). Profiles are shareable as JSON packs so communities can distribute calibration data.

Design intent: treat lens correction not as a single dial but as a surface across focal length × aperture. The user's job is to populate that surface with points; the app's job is to read the surface smoothly. Calibration is a first-class workflow, not a one-time setup.

2.1 Why it matters

  • Real lenses are not uniform. A 24–70mm f/2.8 zoom has dramatically different barrel distortion at 24mm vs 70mm, and vignetting at f/2.8 is much heavier than at f/8. A single-profile correction must average — and therefore mis-corrects most frames.
  • Calibration improves with use. Unlike most edits, a lens profile gets better the more the user shoots with that lens. Every photo at a new (FL, aperture) is an opportunity to fill in the surface.
  • Community scales calibration. Once one user calibrates a popular lens, sharing the pack saves every other owner hours of work.

2.2 Terminology

  • Lens entry — one physical lens in the user's library. Identified by make + model (and optional serial / variant notes).
  • Lens profile file — a file in the user's lens folder holding one lens entry. Authored in the Lens module (§7 / §8). Referenced (copy or link) by the Edit lens-correction panel (§4), the same way a node-sheet file is referenced by a layer in Edit. A profile pack (below) is the multi-lens export / import container.
  • Profile point — a single calibration sample at a specific (focal length, aperture) pair, holding correction values for all four parameter types.
  • Profile surface — the implicit 2D function over (focal length, aperture) defined by the lens's profile points and the interpolation rule. What gets applied to a photo.
  • Calibration — the act of creating or refining a profile point by adjusting sliders against a reference image.
  • Profile pack — a JSON file containing one or more lens entries for export / import.
  • Active lens — the lens entry the corrector is currently using for the loaded photo (auto-detected or manually chosen).
  • Coverage — a visual indicator of how well the profile surface is sampled across the (FL, aperture) range the lens actually supports.

3.3 Distortion (geometry)

Brown–Conrady-style radial polynomial correction. Multi-order so the user can correct complex lens curves, not just a single barrel/pincushion term.

FieldTypeRangeNotes
k1numbertypically −0.20 … +0.20Primary radial term. Negative = pincushion correction of barrel; positive = barrel correction of pincushion.
k2numbertypically −0.50 … +0.50Secondary term. Corrects mid-radius residual after k1.
k3numbertypically −1.0 … +1.0Tertiary term. Fine-tuning at extreme radii (wide rectilinears, fisheye-ish). Default 0.
scalenumber0.5 … 1.5Uniform scale applied after un-distort, to keep the output frame size consistent.

UI surfaces k1, k2, k3 as three sliders grouped under "Distortion", and a "Scale" slider beneath them. Reset per slider (double-click label).

3.4 Vignetting (brightness)

Radial brightness compensation. Modelled as a smooth curve the user shapes, not a single multiplier — real lens falloff is rarely a clean radial function.

FieldTypeRangeNotes
strengthnumber0.0 … 2.0Overall correction intensity. 0 = off.
curveCurve—Falloff shape from centre (0.0) to corner (1.0). Same Curve control type as the main editor's vertex curves (Hole #14 in v1 holes). Brightness multiplier on the Y axis.
midpointnumber0.0 … 1.0Where the falloff curve's "knee" sits along the radius.
feathernumber0.0 … 1.0Softness of the falloff edge at the knee.

UI: strength slider, then an editable falloff curve widget, then midpoint and feather sliders.

3.5 Chromatic aberration (lateral)

Lateral (transverse) CA: per-channel radial offset that causes coloured fringes towards the edges. Resolved with a per-channel scale around the optical centre.

FieldTypeRangeNotes
redScalenumber−0.05 … +0.05Red channel radial scale. Negative pulls red inward.
blueScalenumber−0.05 … +0.05Blue channel radial scale. Positive pushes blue outward.
feathernumber0.0 … 1.0How smoothly the offset ramps with radius.

UI: two sliders "R scale" and "B scale" + "Feather". G is implicit (the reference channel).

3.6 Colour vignette (radial colour cast)

Distinct from brightness vignetting: a colour cast that varies with radius (e.g. green corners on older lenses, magenta on some wide-angles).

FieldTypeRangeNotes
strengthnumber0.0 … 1.0How much correction to apply. 0 = off.
huenumber (degrees)0 … 360Colour to push toward (or away from) at the corners.
saturationnumber0.0 … 1.0How saturated the corner cast is.
midpoint / feathernumbers0.0 … 1.0Same role as brightness vignette — where the colour change begins and how softly it ramps.

UI: a colour swatch (clicking opens the standard colour picker per v1 Hole #14) + sliders for strength, saturation, midpoint, feather.

4. Edit integration (the lens-correction built-in)

The Lens Corrector ships as two pieces: the Lens module (the authoring / calibration / library tool — see §7 / §8), and a lens-correction built-in edit inside Edit's edit list that references a lens profile file made in the Lens module. This mirrors the node-editor relationship: a sheet file made in Nodes is referenced (copy or link — spec-copy-link) by a layer in Edit; a lens profile file made in Lens is referenced (copy or link) by the lens-correction built-in edit in Edit. Edit never authors lens profiles — it consumes them.

The lens-correction entry is the first built-in in the edit list (Lens Correction → Basic → Colour → HSL/Mix → Tone Curve → Detail, v1 §3). It is a built-in edit, not a node sheet — it has no node graph; instead it references a lens profile file and exposes two basic sliders (Distort, Vignette). It runs as the very first stage of the pipeline (§1), before any other built-in or custom sheet. Visual list position ≠ execution order: it displays at the top of the built-in list and also executes first; the broader principle (customs execute by phase, not by panel position — v1 §6) still holds for every other entry. With no lens profile file referenced, the panel shows an empty / unknown-lens state (Hole #24) until one is picked or auto-detected from EXIF.

4.1 Panel layout

┌─ Lens Correction  (built-in edit) ─────────────────┐
│ [On/Off]  [Auto EXIF ✓]  Profile: … ▾   [+ New ▸]  │
│───────────────────────────────────────────────────│
│  Distort    ────●──────────                        │
│  Vignette   ──────●────────                        │
└───────────────────────────────────────────────────┘

A compact built-in entry, not a rich four-group panel. The header row carries the on/off toggle, the Auto-from-EXIF checkbox, the profile dropdown, and the + New button (switches to the Lens module, §4.5). Below the header are just two basic sliders (§4.4). Everything else — coverage map, full per-parameter calibration, lens library — lives in the Lens module (§7 / §8).

4.2 Header controls

  • On/Off toggle — off = identity passthrough (no correction). Like a layer's enable toggle (v1 §5). Toggles the whole lens-correction built-in edit.
  • Auto from EXIF — checkbox. On (default) = system reads EXIF on photo load and auto-selects the matching lens profile file in the dropdown (§5). Off = user picks a profile manually.
  • Profile dropdown — pick which lens profile file this photo references (copy or link, §4.6). Lists the lens profile files in the user's lens folder; a one-line summary under the selection shows variant (if any) + profile-point count.
  • + New button — switches to the Lens module to author a new lens profile file (§4.5). The lens library manager, pack import / export, and calibration all live in the Lens module — not behind a menu here.

4.3 Unknown-lens state

When Auto from EXIF is on and no lens profile file matches the photo's LensMake + LensModel, the dropdown shows empty / "Unknown lens — create one?" (Hole #24). The user can pick an existing file manually, or hit + New to switch to the Lens module and create a matching profile.

4.4 Basic sliders (Distort & Vignette)

Below the header, the panel exposes just two basic sliders: Distort and Vignette. These are per-photo overrides (§6) on top of the referenced profile's interpolated correction — a quick nudge without leaving Edit. The full per-parameter correction (k1/k2/k3/scale, falloff curve, chromatic aberration, colour vignette) is authored in the Lens module (§7), not exposed here.

  • Distort — a single slider that offsets the profile's overall distortion correction for this photo.
  • Vignette — a single slider that scales the profile's vignetting correction for this photo.
  • Sliders match the main editor's numeric conventions (double-click value to type, double-click label to reset, Alt-drag for fine adjust — v1 §4).

4.5 Add new profile

The + New button in the panel header switches to the Lens module to author a new lens profile file (§7). On creation, the new file is referenced by this panel (copy or link per §4.6). All profile authoring, calibration, and profile-point editing happens in the Lens module — never in Edit.

4.6 Copy / link semantics

  • The lens-correction panel references a lens profile file from the user's lens folder. On adding / swapping a file, the app asks copy or link? — the same prompt as a custom sheet layer (spec-copy-link).
    • Copy — a snapshot of the lens profile file is frozen into the photo's edit data; later edits to the file in the Lens module do not affect this photo.
    • Link — the panel stays live to the lens profile file; editing the file in the Lens module updates this photo's correction. A linked panel whose file is missing / moved shows a broken-link badge (parallels spec-library §5).
  • A lens profile file can set a copy/link default (§3.1) to skip the prompt — the same override mechanism as a sheet file (spec-copy-link). The default is "ask" unless the profile file specifies otherwise.
  • Auto from EXIF (§4.2) auto-selects which lens profile file to reference by matching LensMake + LensModel; the resulting reference follows the same copy / link default. If no file matches, the panel shows the unknown-lens state (Hole #24).
  • Per-photo overrides (§6) live on the photo's edit data, on top of the referenced file's interpolated values — independent of copy / link.

5. EXIF auto-detect & interpolation

5.1 EXIF fields read

EXIF tagUsed for
Make + Model (camera)Sanity check / display.
LensMake + LensModelMatch a lens entry in the library.
FocalLengthTag the active point on the (FL, aperture) surface.
FNumberTag the active point on the (FL, aperture) surface.
LensSerialNumber (if present)Disambiguate multiple copies of the same lens model.

5.2 Match algorithm

  1. Read LensMake + LensModel from the photo.
  2. Find lens entries where make + model match (case-insensitive, trimmed).
  3. If multiple matches, prefer the one with a matching LensSerialNumber; otherwise prompt the user to pick (Hole #25).
  4. Read the photo's FocalLength and FNumber → these define the (FL, aperture) coordinate on the lens's profile surface.
  5. Compute the interpolated correction (see §5.4).
  6. Apply it as the first pipeline stage (§1).

5.3 Missing EXIF

If the photo lacks LensModel or FocalLength (scanned images, heavily cropped JPEGs), the corrector falls back to manual mode and shows a notice: "EXIF incomplete — choose lens and (FL, aperture) manually." See Hole #26.

5.4 Interpolation rule

When a photo is at a (FL, aperture) coordinate that doesn't exactly match a profile point, the corrector finds the nearest profile points in (FL, aperture) space and linearly interpolates their parameter values. Linear is the chosen model: simple, predictable, and behaves well with sparse data — a user with one or two points gets sensible corrections, and the surface refines gracefully as more points are added.

5.5 How neighbours are found

  1. Compute the photo's (FL, aperture) coordinate.
  2. For each profile point, compute a distance. Default metric: weighted Euclidean with weights that emphasise focal length — empirically, distortion varies more sharply with zoom than with aperture. 
      d = sqrt( ( (FL - FL_p) / FL_range )² + ( (A - A_p) / A_range )² )
  3. Take the two nearest profile points (or one if only one exists; or zero if none exist).
  4. Weight each by inverse distance (with epsilon against division-by-zero at exact match).
  5. Linearly combine the per-parameter values: distortion, vignetting, CA, colour vignette — each interpolated independently (no cross-parameter coupling).

5.6 Out-of-range behaviour

If the photo's (FL, aperture) is outside the convex hull of the lens's profile points, the corrector uses the single nearest profile point with a "Far from calibration" badge on the panel (no extrapolation). See Hole #27 for the visual treatment.

5.7 Coverage indicator

The coverage heatmap lives in the Lens module (calibration screen §7 / library manager §8), not in the compact Edit panel. It shows density of profile points across (FL, aperture) for the active lens:

  • Empty cells = no nearby points. Photos at those coordinates will use the nearest point only.
  • Dense cells = many points; interpolation is most accurate here.
  • Current photo's cell is highlighted so the user can see at a glance how well-supported the current shot is.

The indicator's job is to motivate the user to fill gaps in the surface.

6. Per-photo overrides

Even with a well-calibrated surface, a particular photo may need a tweak (a focus shift, an off-calibration sample). The two basic sliders in the Edit panel (§4.4 — Distort, Vignette) act as per-photo overrides on top of the referenced profile's interpolated correction. The full per-parameter surface (k1/k2/k3, falloff curve, CA, colour vignette) is edited in the Lens module on the profile file, not per-photo in Edit.

6.1 How overrides combine with interpolation

For each parameter, the corrector stores a per-photo override value (default = identity / "use interpolation"). The final value applied is:

final = interpolated(profile surface) ⊕ per-photo override

where ⊕ is parameter-appropriate combination:

  • Distort slider — scales the profile's overall distortion correction for this photo (additive offset on the combined k1/k2/k3/scale result).
  • Vignette slider — scales the profile's vignetting correction for this photo (multiplicative; a 0.1 nudge = 10% more / less correction).

CA and colour vignette are not per-photo adjustable in this version — they follow the referenced profile's interpolated values exactly. Exposing them per-photo is deferred.

6.2 Override affordances

  • When a slider is at its default (no override), it shows in the "interpolated" state — no badge.
  • When the user moves it, a small badge appears next to the slider label: "overridden".
  • Double-clicking the label resets the override to "use interpolation" (back to interpolated state).
  • A "Reset all overrides" link at the bottom of the panel clears every per-photo adjustment in one click.

6.3 Storing overrides

Per-photo overrides live on the photo's edit metadata, not in the lens entry. Editing the lens profile does not overwrite a photo's overrides, and removing a photo's overrides does not affect the lens profile. They are independent layers of state.

7. Calibration workflow (Lens module)

Calibration is the main workspace of the Lens module (the way the node editor is the Nodes section, v1 §7). Entered from the Lens header, or from the Edit lens-correction built-in edit's + New button (§4.5, which switches to the Lens module).

7.1 Entry points

  • + New (Edit lens-correction built-in edit, §4.5) — switches to the Lens module and starts a new lens profile file; if a photo is loaded, calibration opens against it at the current (FL, aperture).
  • Lens header — works with no photo loaded; opens calibration against a neutral test pattern (see §7.3). Also the entry point for editing an existing profile or adding points to an existing lens.
  • Lens library manager (§8) — calibrate-any-existing-point picker.

7.3 Reference image

Calibration can be run on:

  1. The loaded photo — if a photo is loaded, the canvas shows it. Best for "I have a shot of a grid at 24mm f/2.8, let me dial in corrections for that exact (FL, aperture)."
  2. A neutral test pattern — if no photo is loaded (or the user explicitly wants a clean reference), the app shows a synthetic built-in grid pattern: regular orthogonal lines on a neutral grey background, filling the frame. The user adjusts sliders against this known geometry, which is the most reliable way to correct distortion.

Either way, the canvas applies the current corrections in real time so the user sees the grid undistort / brighten as they adjust.

7.4 Grid overlay

A separate overlay layer the user can toggle on/off and adjust opacity for. Two modes:

  • On top of the photo / test pattern — fine orthogonal lines the user aligns against known straight features (door frames, horizons, architectural edges in the photo).
  • Standalone — the full grid is the canvas, used as the reference in §7.3 option 2.

Controls: toggle, opacity slider, and a spacing dropdown (fine / medium / coarse) for the grid line frequency.

7.5 Calibrating each parameter

The four correction groups behave differently during calibration:

GroupHow the user calibrates
DistortionAdjust k1/k2/k3 until grid lines in the photo / test pattern are perfectly straight. Distortion is the easiest to "see" against a grid.
VignettingAdjust strength + shape the falloff curve until corner brightness matches centre brightness on a uniform subject. The curve widget is the primary control here.
Chromatic aberrationZoom in on high-contrast edges near the corners. Adjust R/B scale until the colour fringes disappear.
Colour vignetteAdjust hue + saturation + strength until the corners are colour-neutral.

The user can work through the groups in any order. The Save button at the bottom commits the current values as a new profile point (or updates the existing one being edited).

7.6 No-photo / no-lens state

Same principle as the node editor (v1 §10.5): the user can author profile points with no photo loaded. In that case, the canvas shows the built-in grid pattern at a neutral focal length / aperture, and the user must set the (FL, aperture) explicitly in the right-panel settings before saving. No "sample image" concept — just the grid.

8.2 Per-lens actions

  • Expand → show all profile points with their (FL, aperture) and source.
  • Edit → open calibration for the closest point to the current photo, or a chosen point.
  • Add profile point → open calibration with a blank new point at the current photo's (FL, aperture).
  • Rename / change variant / change range.
  • Duplicate (for "I bought a second copy of the same lens, calibrate separately").
  • Delete — with confirm; warns if any photo currently uses it.
  • Export pack — writes a JSON pack containing this lens (or all lenses — see §9).

8.3 Per-profile-point actions

  • Edit → open calibration on this point.
  • Duplicate (e.g. "duplicate at 35mm f/2.8 → set to 35mm f/4").
  • Delete — with confirm.
  • Inspect raw values → a popover showing the JSON for this point (Hole #28).

8.4 Search & filter

Filter-as-you-type across make + model + variant + notes. Lens entries that match show with all their points; non-matches collapse away. An empty search shows everything.

9. Profile packs & JSON format

Lens data can be exported as JSON packs and imported from packs. This is how communities share calibration data.

9.1 Export

  • Per-lens export — one lens, all its points. File: {lens-slug}.lenspack.json.
  • Whole-library export — all lenses, all points. File: all-lenses.lenspack.json.
  • Selective multi-lens export — checkboxes in the library manager; file: {n}-lenses.lenspack.json.

9.2 Import

  • File picker accepts one or more .lenspack.json files.
  • For each lens in the pack, the app compares make + model + variant against the local library:
    • No match → import as a new lens entry.
    • Exact match (same make + model + variant) → merge profile points: points whose (FL, aperture) is new are added; existing points are left untouched unless the user picks "overwrite" or "merge newer-wins" (see Hole #29).
  • Author and notes from the pack are preserved as metadata on the lens entry; users can see "sourced from hugh's pack, 2026-06-10".

9.3 Pack provenance

Every pack carries an author field and per-point calibratedFrom + notes. The lens library surfaces this in the manager (§8) so users know where a profile came from. The point of a community is knowing which calibrations to trust.

9.7 Versioning

Format version is bumped on backward-incompatible changes only. The app refuses to import a pack whose version is higher than it understands, and warns (but accepts) on packs whose version is lower (best-effort load, missing fields use defaults). See Hole #30.

Holes #24–#39 (Lens Corrector)

#24 — Unknown-lens state in the panel high resolved

Resolved by §4.3. When Auto from EXIF is on and no lens profile file matches, the dropdown shows empty / "Unknown lens — create one?"; the user can pick an existing file manually or hit + New to switch to the Lens module and create a matching profile.

#25 — Multiple matches on LensMake + LensModel high resolved

Resolved by §5.2 step 3. If multiple lens entries match LensMake + LensModel and no LensSerialNumber disambiguates, the app prompts the user to pick. Remaining detail (whether and how the choice is remembered per-photo) is implementation polish, not a blocking gap.

#32 — Coverage heatmap resolution med

§4.1 / §5.7 describe the coverage heatmap, which lives in the Lens module (not in the compact Edit panel). Confirm: the cells are binned (e.g. one cell per (f-stop × 5mm FL bin)), not continuous. #33 — Vignetting curve interpolation across points med

The vignetting curve (§3.4) is a control curve — it has its own point list. When interpolating between two profile points at different (FL, aperture), the curves have potentially different point counts. Confirm: interpolation samples each curve at a fixed set of x-values (e.g. 0, 0.1, 0.2, …, 1.0), linearly interpolates the y-values, and reconstructs a curve from the combined samples.
