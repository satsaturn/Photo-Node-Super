Photo Editor — v1 Spec

The active build target. v1 is a personal project — no installer, no distribution, one user. This spec is self-contained. The original combined spec lives in spec.html (historical reference). Each deferred feature has its own extension spec, listed in §16.

v1 scope summary. Library (basic: import + grid + filmstrip only) + Edit (global edit list only; mask and crop tabs are disabled placeholders) + Nodes (full authoring). The Lens tab in the header is a disabled placeholder. A gear icon in the header opens an empty Settings dialog (§15). Custom sheets are linked-only (no copy snapshot, no copy/link prompt). The canonical mockup for the Edit module view is docs/mockups/final-mockup.html; other modules share the same shell, header, and filmstrip but have their own module bodies.

1. Core concept

A Lightroom-style non-destructive photo editor where every edit group is a node sheet — a saved node graph. The key differentiator: each sheet contains a UI node whose vertices are the controls shown in the main editor, and each vertex pushes its value to every node parameter wired to it — so one slider can drive many nodes.

2. Terminology

  • Node sheet / sheet — a saved node-graph file in the user's sheets folder. Reusable. Has a phase property (see §6).
  • Layer — an instance of a sheet placed into an edit list (global or mask).
  • UI node — special node in a sheet; its vertices are the user-facing controls.
  • Vertex — one control on the UI node; its value flows out along all connected edges.
  • Edit list — an ordered container of layers. Two kinds: the global edit list (one per photo) and each mask's edit list (deferred — masks are not in v1; see spec-masks). Same structure & rules for both.
  • Built-in sheet — a locked sheet shipped with the app (graph not user-editable). Defines phase landmarks globally.
Deferred to spec-lens: the "built-in edit" term (a non-sheet entry like Lens Correction) is not used in v1 — the Lens Correction built-in slot is reserved/empty in v1.

3. Main editor layout

The Edit module has no left panel. The full centre is a single photo canvas showing the current-result view. No before/after in v1 (§9). The module view follows the v1 visual language described in §11 — light grey canvas with a grain texture, dark UI surfaces floating on top, 12px gaps between the image and every floating element.

  • Top toolbar — zoom dropdown (Fit, 25%, 50%, 100%, 200%) + recenter button (crosshair icon; refits the image to the current available canvas area — the only way to recover the default fit after manual pan/zoom) + full-screen button.

Right panel — two-card floating layout

The right side of the Edit module holds two independently-positioned cards stacked vertically: a histogram card on top and an edit-panel card below it. Both float over the canvas, and the 12px gap rule (§3.1) is honoured at every resize.

  • Histogram card — per-channel + luminance, read-only. Two-box structure: a small header pill labelled "HISTOGRAM" and a dark canvas showing the curve. Resizable height 130–360px via a bottom handle; the edit-panel card's top handle drags the histogram's top edge.
  • Edit-panel card — tabbed container with three tabs: global (active in v1), mask and crop (disabled placeholders, greyed out with a "coming in a future version" tooltip). The global tab shows the global edit list for the current photo. Top→bottom = visual order within the list.
  • Built-in edits always present, fixed visual order among themselves (Lens Correction → Basic → Colour → HSL/Mix → Tone Curve → Detail), Lightroom-style. Not removable. The Lens Correction slot is reserved/empty in v1 — it executes as identity until the Lens module ships (see spec-lens). Visual list position ≠ execution order — see §6.
  • Custom Edits section pinned at the bottom with a + button.
  • Phase-group dividers. Within the Custom Edits section, layers are visually grouped by their execution phase (§6). A thin divider labelled with the phase name (e.g. "After exposure", "At end") separates each group. Dividers are only shown for phases that currently have at least one layer — empty phases are hidden. Layers cannot be dragged across a divider; reordering is constrained to within the same phase group. To change a layer's phase, edit its sheet settings (§12.2.1).

Pressing + creates a custom layer header containing:

  • New file button → authors a fresh sheet (opens node editor)
  • Dropdown of existing sheets in the user's folder
  • Picking/creating fills the panel with that sheet's UI-node controls.

In v1, the new layer is added as linked-only — no copy/link prompt is shown. See §5 for the trimmed layer header.

3.1 Canvas & panel interactions

The Edit module view is interactive — the image, the right column, the histogram, the edit panel, and the filmstrip all have drag-to-resize and (for the image) drag-to-pan + wheel-to-zoom gestures. Bounds, gaps, and behaviour are documented here so the build has a single reference.

  • Image pan. Click-drag on the image moves it; cursor grab → grabbing. Implementation uses transform-origin: 0 0 and a translate(x,y) scale(s) transform; the initial cursor offset is captured on mousedown to avoid jitter.
  • Image zoom. Mouse wheel over the image, 1.1× per tick, clamped 0.05–20×. Zoom centres on the cursor — the point under the cursor stays stationary in image space. Default fit never scales the image up (fit scale ≤ 1).
  • Fit-to-canvas. Computes the available rectangle from the live getBoundingClientRect() of the toolbar (top boundary), right column (right boundary), filmstrip (bottom boundary), and a 12px gap from the left edge; centres the image within that rectangle. Called on initial load and on Recenter.
  • Right column width. Draggable via a left-edge handle on the right column. Clamped 240–640px. Dragging also updates the filmstrip's right offset to columnWidth + 24 to keep the 12px gap.
  • Histogram height. Draggable via a bottom handle on the histogram card. Clamped 130–360px. The edit-panel card's top handle also resizes the histogram — it drags the histogram's top edge (uses −dy so drag-down makes the histogram shorter).
  • Filmstrip height. Draggable via a top handle. Clamped 74–200px. Frame size scales with the strip height: frameH = max(48, stripH − 26), frameW = frameH × 1.5 (3:2 aspect maintained). Minimum frame height 48px.
  • Filmstrip right handle. A secondary control for the right column's width (same 240–640 clamp, same columnWidth + 24 formula for the filmstrip's right offset). The right column's own left handle and the filmstrip's right handle are equivalent — drag either to resize the right column.
  • 12px gap rule. The image never touches any floating element. Fit-to-canvas and every resize gesture honour the 12px gap; the right column's left edge sits 12px to the right of the image, the toolbar's bottom edge sits 12px above it, and the filmstrip's top edge sits 12px below it.

4. UI node & controls

  • A vertex is a control. Control types: Number, Color, Curve, Boolean, Dropdown/enum.
  • Wire a vertex to one or more node params; the control's value is pushed to all of them (e.g. one "Warmth" slider feeding both Temperature and Tint).
  • Authoring controls per vertex: custom label, custom range, reorder within the panel.
  • Grouping = shared vertex (one vertex → one control driving N params). No separate sub-group mechanism.
  • Numeric conventions (Lightroom defaults): double-click value to type, double-click label to reset, Alt-drag for fine adjust.

5. Layer header controls

Every layer in v1 exposes in its header:

  • Enable/bypass toggle — off = identity passthrough (Image-In → Image-Out, controls ignored).
  • Reorder handle — drag within a phase group (sets tie-break execution order within that phase bucket — see §6). The handle cannot drag a layer across a phase-group divider (§3); to move a layer to a different phase, edit its sheet settings.
  • Edit button → opens the node editor for that sheet.
Deferred to spec-copy-link:
  • Sheet-file picker — dropdown to swap which file this layer points to. In v1 a layer is permanently linked to one sheet; to swap, remove the layer and re-add.
  • Pin icon → global panel slot (stays visible across all photos & restarts; slider values still stored per photo).

6. Pipeline execution order

Customs always display at the bottom, but execution is by phase — a property of the sheet file. Four phase buckets, with built-ins defining the boundaries:

OrderBucketContents
0Lens correctionReserved in v1 — identity passthrough. Lens Correction built-in slot ships empty; the Lens module and its profile files are deferred (see spec-lens).
1CropDeferred in v1 — identity passthrough. Crop / straighten / flip → spec-crop.
2First / at startcustoms with phase=first
3(built-in)Basic — Exposure, Contrast, Highlights, Shadows, Whites, Blacks
4After exposure correctioncustoms with phase=after-exposure
5(built-in)Colour (Temp/Tint/Vibrance/Saturation), HSL / Colour Mix, Tone Curve
6After colourcustoms with phase=after-colour
7(built-in)Detail — Sharpen, Luminance + Colour Noise
8At endcustoms with phase=at-end
  • Tie-break within a bucket: panel position within the Custom Edits section (drag to reorder = sets execution order among same-phase sheets).
  • Built-ins are fixed in both panel and pipeline; their positions are the phase landmarks.
  • The 4 buckets exist in every edit list even if a built-in is absent — an empty slot just means its bucket has no built-in; customs still run in bucket order.
Deferred to spec-copy-link: phase-lock-on-file (a linked layer's phase is locked to the file) and copy-vs-link phase-freeze semantics. In v1 all layers are linked-only with no copy snapshot, so phase is always the file's phase — no freeze complexity.
Deferred to spec-lens: §6.1 Lens-correction pipeline integration (lens correction as atomic stage 0 with interpolation/edge cases). v1 ships with the Lens Correction bucket as identity.

7. Node editor (authoring) — the Nodes section

  • A header section (Nodes), not a full-screen toggle inside Edit. Entered from the header, or via a layer's Edit button in the Edit module (switches to Nodes with that sheet loaded). Works with no photo loaded.
  • No-photo state: blank graph; header ops let you open an existing sheet or start new. Image-In shows nothing; thumbnails stay blank until the sheet is applied to a real photo. No sample-image concept exists.
  • Canvas: per-node output thumbnails on each node body + main canvas shows final sheet output (on Image-Out).
  • No nested sheets. A sheet is a flat graph — a node cannot reference or embed another sheet file. Composition is achieved by stacking layers in a photo's edit list (§6), not by nesting graphs. This constraint may be revisited in a later version.
  • Built-in sheets are locked — openable for inspection/thumbnail-scrub but graph not editable.
  • Error states. If a node fails during evaluation (bad params, unsupported bit depth, evaluation error): the node passes its input image through unchanged, its body border turns red, and a warning badge appears on its header. Hovering the badge shows a tooltip with the error message. Downstream nodes receive the pass-through and continue. If Image-Out itself fails, the main canvas shows a "Render error" message. Errors are also logged to the developer console.

8. History / undo

  • Per-sheet undo: each layer keeps its own undo stack (slider drags + graph edits for that sheet), not one global timeline.

10. Node graph layout

A sheet is a single canvas holding two coexisting planes, distinguished by edge type/colour.

10.1 Image plane (edges carry Image)

  • Image-In — terminal source node. Represents the photo arriving at this sheet's phase. Shows the input image as its on-node thumbnail.
  • Effect nodes — each has an Image-In port, an Image-Out port, plus param ports. The chain may branch and merge (e.g. split into a luminance chain + a colour chain, recombine via a Blend node).
  • Image-Out — terminal sink node. The sheet's final result; its thumbnail is what the main canvas displays.
  • Arity: exactly one Image-In and one Image-Out per sheet (v1).
  • Topology: acyclic (DAG) — non-destructive editing is feed-forward.

10.2 Control plane (edges carry Number | Color | Curve | Bool | Enum)

  • UI node — zero or one per sheet (zero = fixed transform, no controls). Its vertices are the controls shown in the main panel. Each vertex is typed. A vertex is the source of its value; the value pushes out along all connected edges to effect-node params. No on/off switch on the UI node itself.
  • Math/utility nodes (minimal v1 set): Number constant, Math (add/mul/sub/div), Remap/clamp, Curve-from-number, Color constant. These sit between UI vertices and effect params so a slider's value can be sculpted before reaching a node.
  • Effect-node param ports are sinks. A param is either bound (driven by a UI vertex, possibly via math nodes) or free (uses the node's own default, set inside the node editor).

10.3 Binding gesture & semantics

  • Drag from an effect node's param port to the UI node → creates/attaches a typed vertex matching the param's type and binds that param to it.
  • One vertex may bind multiple params (one control → many nodes).
  • A bound param ignores its node default; an unbound param uses its node default.
  • Vertex authoring controls (per §4): custom label, custom range, reorder within the panel.

10.4 Bypass

The layer's enable/bypass toggle (header, §5) makes the sheet act as identity: Image-In passes straight to Image-Out, controls and graph ignored.

10.5 Authoring with no photo

When the Nodes section is opened with no photo loaded, the graph starts blank; Image-In shows nothing (or a neutral placeholder), and effect-node thumbnails + the final canvas render stay blank until the sheet is applied to a real photo. Users author by topology and parameter values alone, then see results once the sheet is placed on a photo. No sample-image concept exists.

11. Visual language

v1 has one theme (light grey with a grain texture) and a small set of geometry / typography conventions shared by every module view. The data-type palette and node-header tints in §11.1–11.3 then add colour-coding on top of this baseline for the node editor.

v1 default theme & surface style

The v1 build target is a light grey theme with a grain texture.

  • Canvas background — light grey #e8e8ea with a grain overlay (SVG feTurbulence, 240×240 tile, opacity 0.6, dark color matrix). The grain is the background; no separate background image.
  • Floating pills (header, toolbar, tabs, icon cells) — solid #2a2a30 (--hdr-pill). Used for chrome that sits above the canvas with no grain showing through.
  • Floating cards (histogram, edit panel, section, custom layer, filmstrip) — translucent rgba(42,42,48,0.75) (--panel) so the grain shows through. 1px border in --panel-border (#4a4a52).
  • Section header / tabs background — --panel-section (#34343c), one shade lighter than cards, for a clear header strip inside a card.
  • Accent — --accent (#6ea8ff) for active tabs, the layer enable toggle, the slider knob, and the active filmstrip frame border.

Geometry convention

  • Pills (tabs, dropdowns, toolbar buttons, icon cells) — border-radius: 999px.
  • Cards (histogram, edit panel, section, custom layer, filmstrip) — border-radius: 12px.
  • No shadows anywhere in the v1 build.

Typography

  • Font stack: -apple-system, "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif.
  • Tabs pill: 11px, 0.08em letter-spacing, uppercase.
  • Section / layer header: 12px, weight 600, --panel-text on --panel-section.
  • Slider label: 11px --panel-text-2; slider value: 10px --panel-text-3 with tabular-nums.

12.1 Left — canvas zone

  • Canvas toolbar: zoom in/out, fit-to-view, 1:1; thumbnail toggle (show/hide per-node output thumbnails).
  • Graph canvas holds the two planes (image + control), per §10. Image-Out's thumbnail is the final sheet output shown on the main canvas when not in the node editor.

12.1.1 Canvas interactions

Provided by React Flow (§14.2). v1 adopts the following conventions:

  • Pan — middle-click drag, or Space + left-click drag. Cursor changes to grab → grabbing.
  • Zoom — mouse wheel or trackpad pinch. Clamped 0.05–20×. The fit-to-view button in the toolbar resets zoom to fit all nodes.
  • Drag-to-connect — drag from a source port (right side of node) to a target port (left side). Invalid connections (type mismatch, would create a cycle) show a red indicator and are rejected on drop. Valid connections snap into place.
  • Delete edge — click to select an edge (highlights), press Delete or Backspace.
  • Reconnect edge — drag either endpoint of a selected edge to a different port.
  • Select node — click a node to select it (single selection). Click on empty canvas to deselect.
  • Multi-select nodes — Shift+click to toggle a node into/out of the current selection; or drag a selection rectangle (marquee) on empty canvas to select all nodes within.
  • Delete nodes — select one or more nodes and press Delete or Backspace.
  • Copy / paste nodes — select node(s) and press Ctrl+C / Ctrl+V. Pasted nodes are offset 30px down-right from the originals. Edges that connected two copied nodes are preserved; edges to/from uncopied nodes are dropped.
  • Undo / redo — Ctrl+Z / Ctrl+Y per the per-sheet undo stack (§8).
  • Drag move nodes — standard click-drag on the node body to reposition. Arrow keys nudge the selected node by 1px (or 10px with Shift).
  • Select all — Ctrl+A selects all nodes on the canvas.
  • Fit to view — F key centres the canvas and zooms to encompass all nodes (same as the toolbar button).
  • Quick-add node search — Tab opens an overlay search box at the canvas centre. Type to filter nodes by name across all palette categories; pressing Enter adds the top result at the current cursor position. Esc closes the overlay without adding.
  • Escape — deselects all nodes and closes any open popover (properties panel returns to sheet settings).

12.2 Right top — contextual properties panel

The right-top panel switches content based on what is selected on the canvas. The palette always stays visible in the right-bottom section (§12.3).

  • Nothing selected (default) — shows sheet settings (§12.2.1).
  • Node selected (effect / terminal / math) — shows node properties (§12.2.2).
  • Vertex selected (on the UI node) — shows vertex properties (§12.2.3).

12.2.1 Sheet settings (default, nothing selected)

  • Sheet name — editable; shown in the main-editor layer header and the sheet-file dropdown.
  • Name collisions. If two saved sheets have the same display name, the dropdown disambiguates by appending the filename in muted text: "Sunset Tone" — sunset-tone.json. On Save As, if the target filename already exists, a "Replace existing file?" confirmation is shown before overwriting.
  • Execution phase — dropdown: First / After exposure / After colour / At end (§6).
  • Description / notes — free text; shown as a tooltip in the sheet-file dropdown so users recall what a saved sheet does.
  • Author / last-saved timestamp — for managing a growing library.
  • Header ops (from §7): Save, Save As, Rename, Duplicate, Delete — applied to the open sheet.
  • Note — Built-in duplication. Duplicating a locked built-in sheet produces a new unlocked sheet file with an identical graph. The user is prompted to name the copy (default: "{original name} Copy"), and it is saved to the user's sheets folder. The original built-in remains unchanged.
Deferred to spec-copy-link: the "Copy/Link default override" sheet setting (v1 has no copy, so no per-sheet override is needed).

12.2.2 Node properties (when a node is selected)

  • Node name — editable text field. Defaults to the node type (e.g. "Exposure", "Blend").
  • Category — read-only indicator: Terminal / Effect / Math-utility / UI.
  • Description — optional free-text notes for the author.

Free-param values are edited directly on the node body (§12.3.2), not in this panel.

12.2.3 Vertex properties (when a vertex on the UI node is selected)

  • Label — editable text; this is what appears as the control's name in the main-editor panel.
  • Type — read-only: Number / Color / Curve / Boolean / Dropdown. Set on creation.
  • Range (Number only) — min, max, step value. The main editor's slider uses these bounds.
  • Default value — the value a new layer gets before the user touches the control. Expressed per type: a number, a colour swatch, a curve, a boolean toggle state, or a dropdown index.
  • Dropdown options (Dropdown/enum only) — an editable list of option labels. The user adds, removes, and reorders items; each item becomes one entry in the main editor's dropdown.
  • Reorder — up/down buttons to change the vertex's position in the main editor's control list. The order in the UI node determines the top-to-bottom order in the edit panel.

12.3 Right bottom — node palette

  • Search box at the top — filter-as-you-type across node names.
  • Collapsible category sections below:
    • Terminals — Image-In, Image-Out. Shown but locked; each sheet has exactly one of each, and they cannot be added or removed.
    • Effects — the 13 effect nodes listed in §12.3.1. These are the building blocks for custom sheets.
    • Math-utility — Deferred. The category header is shown but collapsed by default and marked "Coming in a future version." No nodes ship in v1. The planned set (Number constant, Math, Remap/clamp, Curve-from-number, Color constant) is listed in §10.2 for reference.
    • UI — "UI Node" entry. Greyed out if a UI node already exists on the canvas (a sheet may have zero or one UI node).
  • Add gesture: drag a node from the palette onto the canvas, OR click to drop it at the canvas centre / last-click position.
  • Categories collapse to save vertical space; search filters across all categories regardless of collapse state.

12.3.2 Free-param editing on node bodies

When a node param is free (not bound to a UI vertex), its default value is set directly on the node body. The editing widget sits next to the socket label, Blender-style. When a param is bound (connected to a UI vertex via §10.3), the widget is hidden — the value is driven by the connected vertex.

  • Number — small editable number field. Click to type a value; drag horizontally to scrub. Shows 1–3 decimal places as appropriate.
  • Color — colour swatch thumbnail. Click opens a colour picker popover (RGB sliders + hex input).
  • Curve — small curve preview thumbnail. Click opens a curve editor popover with control-point editing.
  • Boolean — inline toggle / checkbox, flush with the socket row.
  • Dropdown/enum — compact inline dropdown selector, same row as the socket label.
  • Node resize: if the body has more params than fit, the node shows a scrollbar. The user may also drag-resize the node vertically to see more params at once.
  • Collapsible param groups: nodes with many params (e.g. HSL's 8 colour bins) group them under collapsible headers. Clicking the header triangle toggles visibility of that group's fields on the node body.
Blender reference. The free-param widget sits on the same row as the socket label, left-aligned after the label text. The socket dot is always visible on the left edge of the node; connecting a wire to the socket hides the inline widget. v1 follows this pattern precisely.

13. App shell & Library module

v1 Library scope: import + browse + select — the minimum to get photos into the app and pick one to edit. Everything else is deferred to spec-library.

13.1 Catalog & storage model

  • A catalog (a SQLite file in app-local storage) is the source of truth for: the list of known photos, each photo's on-disk path, metadata (EXIF), the per-photo edit list (§2/§3, global), and the preview/thumbnail index.
  • Photo files stay on disk where the user put them — referenced in place, never copied on import. The catalog stores an absolute path plus a light identity tuple (file size + mtime) to detect moves/mismatches.
  • Previews — one thumbnail and one standard preview per photo — are generated at import and cached in app-local storage (not as DB blobs). The full-resolution file is read on demand by the Edit module for canvas rendering.
  • Edits live in the catalog DB as each photo's edit list. One catalog per user profile in v1; the DB file can be copied manually for backup (multi-catalog support is deferred — §9 / spec-library).
Deferred to spec-library: the optional content-hash duplicate-detection check, Write / Read XMP sidecars, and multi-catalog backup/restore UI.

13.2 Module switcher (Library | Edit | Nodes | Lens)

  • A top-level Library | Edit | Nodes | Lens switcher sits at the top-right of the window (Lightroom-style). The four sections are peer modules in one app, switched via the header — none is nested inside another.
    • Library — browse / import (§13.3).
    • Edit — the photo editor from §3. Right panel = a tabbed container with global, mask, and crop tabs. Only global is active in v1; mask and crop are disabled placeholders. Centre = photo canvas.
    • Nodes — the node editor (§7 / §12): the authoring tool for sheet files. Reached from the header, or from a layer's Edit button (switches to Nodes with that sheet loaded).
    • Lens — disabled placeholder in v1. The lens-corrector tool (calibration / library / packs) is deferred to spec-lens.
  • File relationship for custom sheets: a sheet file made in Nodes is referenced by a layer in Edit. In v1 the reference is linked-only (no copy snapshot). Edit never authors files — it consumes them.
  • Switching sections preserves the current photo and the current selection. Opening a photo in Edit = double-click in Library → switch to Edit with that photo loaded.
  • Bottom filmstrip is shared across all four sections and shows the current source/selection; clicking a filmstrip frame loads that photo into the active section.

13.2.1 Filmstrip behaviour

  • Height — user-resizable via a top handle on the strip, clamped 74–200px. Frame size scales with the strip height: frameH = max(48, stripH − 26), frameW = frameH × 1.5 (3:2 aspect maintained). Minimum frame height 48px.
  • Right-edge handle — a secondary control for the right column's width (same 240–640 clamp as the right column's own left handle, see §3.1). Dragging it updates the filmstrip's right offset to columnWidth + 24 to keep the 12px gap. The right column's left handle and the filmstrip's right handle are equivalent — drag either to resize the right column.
  • Click a frame to set it as the active selection (singleton — only one frame is active at a time). The active frame gets the --accent border.
  • Layout — outer container has overflow: visible (so the resize handles aren't clipped) and 12px radius; inner frame list has overflow-x: auto, overflow-y: hidden and matches the 12px radius so scroll corners stay rounded.

13.4 Import flow

  • The Import button (top toolbar) opens the import dialog.
  • Dialog fields:
    • Source folder picker (a single root; a recursive checkbox is on by default).
    • Candidate list — every supported image file under the root, each with an include checkbox (all on by default) and a small size/date preview.
    • Paths already in the catalog are shown greyed with an "already imported" badge and excluded by default.
  • Preview generation — one thumbnail and one standard preview are built per file at import.
  • Progress — a progress bar with per-file status (EXIF scan → thumbnail → preview), a running count, and a Cancel button (cancel stops remaining files; already-imported files stay in the catalog).
  • Supported formats — common still formats (JPEG, PNG, TIFF) plus camera RAW via the platform decoder. The exact format list and RAW coverage are implementation/stack and remain deferred (§9).
  • On completion the new photos appear in All Photos; the most-recently imported photo is selected.

  • Read edits from XMP sidecar checkbox on import.

13.5 Browse & select

  • Single select: click a thumbnail → loads it into the filmstrip selection; double-click → switches to Edit with that photo loaded.
  • Keyboard: arrow keys move focus/selection in the grid; Enter opens in Edit; Esc clears the selection.
Deferred to spec-library: multi-select (Ctrl/Cmd-click, Shift-click), filename search, sort (by Capture/Import/File name ±).

13.6 Relationship to the edit list & sheets

  • The catalog is where each photo's edit list (global, §2/§3) is persisted. The Edit module reads and writes the edit list through the catalog; nothing about the edit list itself changes from §3–§7.
  • Sheet files stay separate — they live in the user's sheets folder (§2) and are not part of the catalog. The catalog stores, for each layer in each photo's edit list: the sheet file path and its phase (§6). In v1 every layer is a link — no copy snapshot is stored.
Deferred to spec-copy-link: the linked-sheet broken-link / versioning UX (a linked layer whose sheet file is missing/moved shows a broken-link badge + relink).

14. Software stack

Implementation details resolved for v1. v1 is a personal project that runs on Windows 10 21H2+ and Windows 11. No installer, auto-update, code signing, or distribution packaging is in scope. The stack (Electron, React, React Flow, better-sqlite3, libvips, LibRaw, SQLite) is cross-platform in principle, and a Linux port is a long-term personal goal — not scheduled, not committed. GPU acceleration is intentionally deferred; the architecture is designed so the renderer can be swapped in a later version without changing the UI, catalog, or saved edit data.

14.1 Desktop shell & UI

  • Electron — cross-platform desktop wrapper (Chromium + Node.js). Provides the native window, file-system access, and OS integration while the UI is built as a web app.
  • React + TypeScript — the main UI framework for panels, sliders, the library grid, and the node editor.
  • Tailwind CSS — utility-first styling.

14.2 Node graph canvas

  • React Flow — provides the node graph canvas, drag-to-connect wiring, pan/zoom, and per-node thumbnails. The visual language and data types in §11 map directly onto React Flow nodes and edges.

14.3 Catalog & storage

  • SQLite via better-sqlite3 — one local catalog file per user profile, exactly as described in §13.1. Stores photo records, paths, metadata, edit lists.
  • Photo files remain referenced in place; previews are cached in app-local storage.

14.4 Image processing (v1)

  • Sharp (libvips bindings) — handles standard formats (JPEG, PNG, TIFF), resizing, colour adjustments, blending, curves, and export.
  • LibRaw — decodes camera RAW files to a working image that Sharp can process.
  • CPU-only in v1. No GPU acceleration in the first version; all effect math runs through Sharp on the CPU.

14.5 Rendering strategy

  • Two render paths:
    • Fast preview path — operates on a downsampled copy of the image while the user is dragging a slider or scrubbing a control, so the UI feels responsive.
    • Full-resolution path — operates on the full source image and is used for the idle editor view and 100% zoom. Triggered when the user releases a control.
  • The main canvas always shows the best available render: full-res when idle, low-res only during active interaction.
  • 100% pixel zoom displays a crop from the most recent full-resolution render.
Deferred to spec-gpu: the renderer abstraction for future GPU support (WebGPU preferred, WebGL 2 fallback, native-addon option). Each effect node currently exposes render(inputImage, params) → outputImage; the abstraction that lets v2 swap in a GPU path without touching UI/catalog/saved data is detailed in spec-gpu.

14.6 Personal-project build model

  • v1 is a personal project. The app runs from a local dev checkout (npm start) or a portable build folder produced by npm run build (or the Electron-builder dir target — produces an unpacked folder, not an installer). No installer, auto-update, code-signing, or distribution packaging is in scope.
  • If the project is ever released beyond the author's own machines, this section is the place to revisit installer choice, signing, and update strategy.

14.7 Windows-specific concerns

  • High-DPI / per-monitor v2 scaling. Electron handles this natively; verify at 100%, 125%, 150%, 175%, and 200% scales, and across monitors with different scale factors (drag the window between them).
  • File dialogs. Native IFileDialog via Electron's dialog.showOpenDialog / showSaveDialog. The app remembers the last-used folder per dialog type (Open, Import Folder, Export) under the catalog directory in the storage locations below.
  • Path handling. The catalog stores canonical absolute paths; the importer strips any \\?\ long-path prefix the OS may insert and normalises for comparison.
  • Keyboard. Standard Windows shortcuts throughout: Ctrl+O, Ctrl+Shift+O, Ctrl+S, Ctrl+Z / Ctrl+Y / Ctrl+Shift+Z, F1, F11, Esc. Alt-key mnemonics on menus. Win+Arrow snap layouts are handled by Windows natively; the app supports arbitrary window sizes from 800×600 up to maximised.
  • Storage locations (Windows). Per the catalog model in §13.1:
    • Catalog DB: %APPDATA%\Photo-Node-Super\catalog.sqlite (roaming profile, per-user).
    • Preview cache and thumbnails: %LOCALAPPDATA%\Photo-Node-Super\cache\.
    • User-authored node sheets: %APPDATA%\Photo-Node-Super\node-sheets\.
    • Logs: %LOCALAPPDATA%\Photo-Node-Super\logs\.
Deferred to spec-windows:
  • Explorer integration: "Open with Photo-Node-Super" shell verb registered for 12 image/RAW extensions; drag-drop from Explorer (single/multi-file/one-level folder recursion); plus the out-of-v1 list (thumbnail provider, preview handler, "Edit" context-menu verb, jump lists).
  • Lens pack storage location (%APPDATA%\Photo-Node-Super\lens-packs\) — arrives with spec-lens.

#1 — Effect-node set for the palette high resolved

#13 — Built-in sheet duplication low resolved

#14 — Curve / Color / Dropdown control rendering low

A Curve vertex shows in the main panel as an editable curve widget — points? Bezier? RGB vs luminance? A Color vertex — swatch + picker (RGB/HSV/hex/native OS)? A Dropdown — how are its options defined in the node editor? All unspecified but low-risk to fill in later.

#15 — Free-param visibility in main editor low

If a param is free, is its value shown anywhere in the main editor, or only inside the node editor? Probably the latter (only bound controls surface), but worth confirming so users aren't confused why a sheet "does something" they can't see a slider for.

#16 — Sheet library manager low

§12.2 (header ops) and §7 cover the sheet's own header operations (Save, Save As, Rename, Duplicate, Delete) applied to the currently open sheet. But there's no way to manage sheets you're not currently editing (rename/delete/duplicate/categorise/organise) except by opening each one. A small library manager is implied but unspecified.

#17 — Reorder-handle scope low resolved

Resolved by §3 and §5: phase-group dividers in the Custom Edits section group layers by phase. Dividers only appear for non-empty phases. Reordering is constrained within a group — the handle cannot drag across a divider. Phase change is done via sheet settings, not drag.

#18 — Nested sheets / sheet-as-node low resolved

Resolved by §7: no nested sheets in v1. A sheet is a flat graph; a node cannot reference another sheet. Composition is by layers in the edit list.

#19 — Sheet name collisions low resolved

Resolved by §12.2.1: dropdown shows "display name" — filename.json for disambiguation. Save As prompts before overwriting an existing file.

#20 — Error states low resolved

Resolved by §7: failed node passes its input through unchanged, shows a red border + warning badge with tooltip. Downstream nodes receive the pass-through and continue. Image-Out failure shows "Render error" on the canvas.

#21 — Keyboard shortcuts low resolved

Resolved by §12.1.1 and §14.7: node-editor shortcuts defined (F = fit view, Tab = quick-add search, Ctrl+A = select all, Esc = deselect/close, plus existing Undo/Redo, Delete, Copy/Paste, arrow nudge, Space+drag pan). Standard app shortcuts (Ctrl+O/S, F11, etc.) are in §14.7.

#22 — Slider tooltips & value display low

Hover a slider in the main editor — show current value? Range? Unit? Lightroom shows value on hover + while dragging. Confirm we match.

#23 — Phase naming vs built-in contents low

Phase bucket is "After exposure correction" but the built-in there is "Basic" which does Exposure and Contrast/Highlights/Shadows/Whites/Blacks. Minor naming mismatch — "After Basic" might read clearer than "After exposure correction." Cosmetic but worth aligning.
