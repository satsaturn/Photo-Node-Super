GPU renderer — extension spec

The renderer abstraction that lets a future version swap in a GPU path (WebGPU preferred, WebGL 2 fallback, native addon option) without changing the UI, catalog, or saved edit data. Not in v1; v1 runs CPU-only via Sharp (v1 §14.4).

Depends on v1 §14.4 (Sharp + LibRaw CPU rendering) and §14.5 (two render paths). v1 already has render(inputImage, params) → outputImage per effect node — this spec documents the abstraction that protects everything above against a future v2 GPU swap.

1. Renderer abstraction for future GPU support

  • Each effect node exposes a single interface: render(inputImage, params) → outputImage.
  • The rest of the app — UI, catalog, undo, saved sheets — talks to nodes only through this interface and never calls Sharp directly.
  • v2 GPU path: the v2 implementation renders effect nodes via WebGPU (preferred, with a WebGL 2 fallback) inside the existing Electron renderer. Chromium's WebGPU gives compute-shader support and direct access to the GPU without leaving the existing process model. If a future version needs features WebGPU cannot express (e.g. vendor-specific extensions or off-process compute), the alternative is a small native addon (Rust + wgpu, or NAPI) that exposes the same render(inputImage, params) → outputImage interface. The interface contract is what protects the node graph, catalog, undo, and saved sheets across both paths.
