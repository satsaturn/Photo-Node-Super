Photo Editor — Usability Spec (v1)

Working draft · "Lightroom-style UI with a node-graph backend that surfaces user-built sliders." Covers the core editor, the 4-way app shell (Library | Edit | Nodes | Lens), the Library module, the Lens Corrector subsystem (profile data model, Edit integration, EXIF interpolation, calibration, library, packs), and the v1 software stack decision in §15.

Note: This is the original combined spec covering all versions. The active v1 build target is spec-v1.html; deferred scope is split across per-feature extension specs linked from spec-v1 §15. This file is kept as a historical reference.

1. Core concept

A Lightroom-style non-destructive photo editor where every edit group is a node sheet — a saved node graph. The key differentiator: each sheet contains a UI node whose vertices are the controls shown in the main editor, and each vertex pushes its value to every node parameter wired to it — so one slider can drive many nodes.

2. Terminology

  • Node sheet / sheet — a saved node-graph file in the user's sheets folder. Reusable. Has a phase property (see §6).
  • Layer — an instance of a sheet placed into an edit list (global or mask).
  • UI node — special node in a sheet; its vertices are the user-facing controls.
  • Vertex — one control on the UI node; its value flows out along all connected edges.
  • Edit list — an ordered container of layers. Two kinds: the global edit list (one per photo) and each mask's edit list. Same structure & rules for both.
  • Built-in sheet — a locked sheet shipped with the app (graph not user-editable). Defines phase landmarks globally.
  • Built-in edit — a built-in entry in the edit list that is not a node sheet (has no node graph). The Lens Correction entry is the only built-in edit in v1: it references a lens profile file (copy or link) made in the Lens section (§18) instead of running a sheet.

3. Main editor layout

Centre — photo canvas

The Edit module has no left panel. The full centre is a single photo canvas showing the current-result view. No before/after in v1 (§10).

  • Top toolbar — zoom dropdown (Fit, 25%, 50%, 100%, 200%) + full-screen button.

Right panel — histogram + global, mask, and crop tabs

The right panel is a tabbed container. At the very top sits a histogram (per-channel + luminance, read-only). Below it are three tabs: global, mask, and crop. The global tab shows the global edit list for the current photo. The mask tab shows the mask list and the edit list of the currently selected mask. The crop tab is a dedicated geometry panel (see §3.1). Top→bottom = visual order within each list.

  • Built-in edits always present, fixed visual order among themselves (Lens Correction → Basic → Colour → HSL/Mix → Tone Curve → Detail), Lightroom-style. Not removable. Lens Correction is a built-in edit, not a node sheet — it references a lens profile file (copy or link) made in the Lens section (§18) and exposes two basic sliders (Distort, Vignette). Visual list position ≠ execution order — see §6 and §6.1.
  • Custom Edits section pinned at the bottom with a + button.

Pressing + creates a custom layer header containing:

  • New file button → authors a fresh sheet (opens node editor)
  • Dropdown of existing sheets in the user's folder
  • Picking/creating fills the panel with that sheet's UI-node controls.

On add, the app asks: copy or link? (copy = safe snapshot; link = live to the sheet file; choice asked each time, unless the sheet sets a Copy/Link default override — see §13.2).

Main canvas: single current-result view. No before/after in v1 (§10).

Partly open: the app shell (4-way header: Library | Edit | Nodes | Lens), the shared filmstrip, and the left-panel role in Library are now defined in §14.2/§14.3. The Edit module layout is resolved above; remaining unspecified items (zoom dropdown options, histogram rendering detail) are deferred to implementation.

3.1 Crop tab

The crop tab holds the photo's geometry controls. It is a dedicated panel, not a layer in the global edit list.

  • Aspect ratio — presets (Original, 1:1, 4:3, 3:2, 16:9, etc.) plus a free-form option and custom width/height inputs.
  • Straighten — angle slider to rotate the crop, with a level/straighten tool activated from the canvas toolbar.
  • Flip — horizontal and vertical flip toggles.
  • Reset — returns crop, angle, and flip to defaults in one action.

The crop boundary is overlaid on the main canvas. Dragging the canvas overlay adjusts the crop; the right-panel controls update in sync. Crop runs at a fixed pipeline position — see §6.

4. UI node & controls

  • A vertex is a control. Control types: Number, Color, Curve, Boolean, Dropdown/enum.
  • Wire a vertex to one or more node params; the control's value is pushed to all of them (e.g. one "Warmth" slider feeding both Temperature and Tint).
  • Authoring controls per vertex: custom label, custom range, reorder within the panel.
  • Grouping = shared vertex (one vertex → one control driving N params). No separate sub-group mechanism.
  • Numeric conventions (Lightroom defaults): double-click value to type, double-click label to reset, Alt-drag for fine adjust.

5. Layer header controls

Every layer, global or mask, exposes in its header:

  • Enable/bypass toggle — off = identity passthrough (Image-In → Image-Out, controls ignored).
  • Sheet-file picker — dropdown to swap which file this layer points to.
  • Reorder handle — drag within the Custom Edits section (sets execution order within a phase bucket — see §6).
  • Pin icon → global panel slot (stays visible across all photos & restarts; slider values still stored per photo).
  • Edit button → opens the node editor for that sheet.

6. Pipeline execution order

Customs always display at the bottom, but execution is by phase — a property of the sheet file. Four phase buckets, with built-ins defining the boundaries:

OrderBucketContents
0Lens correction stage 0Lens Correction built-in edit referencing a lens profile file (copy or link, §18). Distortion + vignetting + CA + colour vignette, interpolated from the referenced file's active lens profile (§19). Runs before crop, all customs, and all other built-ins.
1CropCrop / straighten / flip from the Crop tab (§3.1). Defines the working frame for everything downstream. Not a layer in the edit list.
2First / at startcustoms with phase=first
3(built-in)Basic — Exposure, Contrast, Highlights, Shadows, Whites, Blacks
4After exposure correctioncustoms with phase=after-exposure
5(built-in)Colour (Temp/Tint/Vibrance/Saturation), HSL / Colour Mix, Tone Curve
6After colourcustoms with phase=after-colour
7(built-in)Detail — Sharpen, Luminance + Colour Noise
8At endcustoms with phase=at-end
Lens correction is stage 0; crop is stage 1. Lens correction runs first because geometric distortion changes the image geometry, so anything downstream (cropping, transforms, masks, blur kernels) needs the corrected coordinates. Crop then defines the working frame before any custom or built-in edit runs. The broader rule still holds everywhere else — visual order ≠ execution order (customs execute by phase, not by panel position).
  • Tie-break within a bucket: panel position within the Custom Edits section (drag to reorder = sets execution order among same-phase sheets).
  • Built-ins are fixed in both panel and pipeline; their positions are the phase landmarks.
  • Phase lives on the sheet file ⇒ a linked layer's phase is locked to the file (duplicate the file to run the same sheet in a different phase). A copied layer freezes the phase at copy time.
  • The 4 buckets exist in every edit list even if a built-in is absent — an empty slot just means its bucket has no built-in; customs still run in bucket order.

6.1 Lens-correction pipeline integration

The lens-correction panel in Edit runs as the very first stage of the pipeline (order 0 in the table above), before crop, any built-in sheet, or any custom sheet. Rationale: geometric distortion changes the image geometry, so anything downstream (cropping, transforms, masks, blur kernels) needs the corrected coordinates. Vignetting and CA are pixel-value corrections, but keeping them in the same first stage means the corrector is a single, atomic module the user toggles. The panel references a lens profile file (copy or link, §18.6); a linked panel re-interpolates whenever the file is edited in the Lens section.

The Lens Correction stage is "phase 0". Crop follows at stage 1 (§3.1). The four custom-sheet phases above are unchanged — they describe the custom pipeline, and both lens correction and crop are upstream of all of them. See Hole #31 for how this affects existing copy/link semantics on the first custom phase.

Bypass

The panel's Enable toggle (§18.2) bypasses the entire corrector: the photo enters the rest of the pipeline uncorrected. Identical semantics to a layer's enable toggle in the main editor (§5).

Edge cases

  • No lens matched (EXIF unknown, Auto from EXIF off) → corrector shows an empty state and outputs the image unmodified. User can still pick a lens manually or create one.
  • Lens has zero profile points → corrector shows an empty state and outputs the image unmodified. A "+ New" CTA switches to the Lens section to start calibrating.
  • One profile point → that point's values are used directly (no interpolation).
  • Two or more points → linear interpolation per §19.

7. Masks

  • A mask = a shape (radial/linear/brush/luminance/etc.) wrapping its own edit list.
  • A mask's edit list is structurally identical to the global one: same built-in sheets available, same 4-phase system, same +/-/dropdown/new-file flow.
  • Masks start empty; user adds only the built-ins/customs wanted for that local adjustment (e.g. local exposure = mask containing the Basic sheet).
  • Principle: a node sheet is identical whether used globally or locally — its phase means the same thing in any edit list, relative to that list's built-ins.
  • Mask-local edits apply only within the masked area.

7.2 Mask list

  • New mask — a dropdown/picker at the top of the mask tab lists the available mask shapes (radial, linear, brush, luminance, depth, subject, etc.). Choosing one creates an empty mask with a default name and adds it to the list.
  • Enable toggle — each mask row has a toggle to enable/disable the mask; disabled masks do not contribute to the result.
  • Name — default names are generated from the shape type ("Radial 1", "Brush sky", etc.). The name is editable inline.
  • Type badge — shows the mask shape (radial / linear / brush / luminance / …).
  • Edit shape — opens the mask-drawing controls for that mask on the photo canvas.
  • Duplicate — creates a copy of the mask including its current shape and edit list.
  • Delete — removes the mask after confirmation.
  • Reorder handle — drag to reorder masks in the list; order affects compositing when masks overlap.

7.3 Selected mask edit list

Selecting a mask in the list loads its edit list beneath the mask list. The edit list has the same structure and rules as the global edit list (§3): built-ins in fixed order, Custom Edits section at the bottom, layer header controls per §5, and the same 4-phase execution rules per §6.

8. Node editor (authoring) — the Nodes section

  • A header section (Nodes), not a full-screen toggle inside Edit. Entered from the header, or via a layer's Edit button in the Edit module (switches to Nodes with that sheet loaded). Works with no photo loaded.
  • No-photo state: blank graph; header ops let you open an existing sheet or start new. Image-In shows nothing; thumbnails stay blank until the sheet is applied to a real photo. No sample-image concept exists.
  • Canvas: per-node output thumbnails on each node body + main canvas shows final sheet output (on Image-Out).
  • Built-in sheets are locked — openable for inspection/thumbnail-scrub but graph not editable.

9. History / undo

  • Per-sheet undo: each layer keeps its own undo stack (slider drags + graph edits for that sheet), not one global timeline.

11. Node graph layout

A sheet is a single canvas holding two coexisting planes, distinguished by edge type/colour.

11.1 Image plane (edges carry Image)

  • Image-In — terminal source node. Represents the photo arriving at this sheet's phase. Shows the input image as its on-node thumbnail.
  • Effect nodes — each has an Image-In port, an Image-Out port, plus param ports. The chain may branch and merge (e.g. split into a luminance chain + a colour chain, recombine via a Blend node).
  • Image-Out — terminal sink node. The sheet's final result; its thumbnail is what the main canvas displays.
  • Arity: exactly one Image-In and one Image-Out per sheet (v1).
  • Topology: acyclic (DAG) — non-destructive editing is feed-forward.

11.2 Control plane (edges carry Number | Color | Curve | Bool | Enum)

  • UI node — zero or one per sheet (zero = fixed transform, no controls). Its vertices are the controls shown in the main panel. Each vertex is typed. A vertex is the source of its value; the value pushes out along all connected edges to effect-node params. No on/off switch on the UI node itself.
  • Math/utility nodes (minimal v1 set): Number constant, Math (add/mul/sub/div), Remap/clamp, Curve-from-number, Color constant. These sit between UI vertices and effect params so a slider's value can be sculpted before reaching a node.
  • Effect-node param ports are sinks. A param is either bound (driven by a UI vertex, possibly via math nodes) or free (uses the node's own default, set inside the node editor).

11.3 Binding gesture & semantics

  • Drag from an effect node's param port to the UI node → creates/attaches a typed vertex matching the param's type and binds that param to it.
  • One vertex may bind multiple params (one control → many nodes).
  • A bound param ignores its node default; an unbound param uses its node default.
  • Vertex authoring controls (per §4): custom label, custom range, reorder within the panel.

11.4 Bypass

The layer's enable/bypass toggle (header, §5) makes the sheet act as identity: Image-In passes straight to Image-Out, controls and graph ignored.

11.5 Authoring with no photo

When the Nodes section is opened with no photo loaded, the graph starts blank; Image-In shows nothing (or a neutral placeholder), and effect-node thumbnails + the final canvas render stay blank until the sheet is applied to a real photo. Users author by topology and parameter values alone, then see results once the sheet is placed on a photo. No sample-image concept exists.

14.1 Catalog & storage model

  • A catalog (a SQLite file in app-local storage) is the source of truth for: the list of known photos, each photo's on-disk path, metadata (EXIF/IPTC), the per-photo edit list (§2/§3, global + per-mask), and the preview/thumbnail index.
  • Photo files stay on disk where the user put them — referenced in place, never copied on import. The catalog stores an absolute path plus a light identity tuple (file size + mtime) to detect moves/mismatches; an optional content-hash check is available at import (§14.4) but not stored as the key.
  • Previews — one thumbnail and one standard preview per photo — are generated at import and cached in app-local storage (not as DB blobs). The full-resolution file is read on demand by the Edit module for canvas rendering.
  • Edits live in the catalog DB as each photo's edit list. A Write XMP sidecars command exports the current edit list to a .xmp-style file beside the original for interop/backup. Read XMP on import is an import-dialog checkbox (off by default) that seeds a photo's edit list from an existing sidecar. Sidecars are never the source of truth; the DB is.
  • One catalog per user profile in v1. Multi-catalog support, catalog switching, and a backup/restore UI are deferred (§14.8); in v1 the DB file can be copied manually for backup.

14.2 Module switcher (Library | Edit | Nodes | Lens)

  • A top-level Library | Edit | Nodes | Lens switcher sits at the top-right of the window (Lightroom-style). The four sections are peer modules in one app, switched via the header — none is nested inside another.
    • Library — browse / organise / import (§14.3).
    • Edit — the photo editor from §3. Right panel = a tabbed container with global, mask, and crop tabs. The global tab shows the edit list: built-in edits (Lens Correction → Basic → Colour → HSL → Tone Curve → Detail) and the Custom Edits section. The mask tab shows the mask list and the selected mask's edit list (§7). The crop tab holds the dedicated geometry panel (§3.1). The Lens Correction built-in edit references a lens profile file made in the Lens section (§18) and exposes basic Distort / Vignette sliders. Centre = photo canvas.
    • Nodes — the node editor (§8 / §13): the authoring tool for sheet files. Reached from the header, or from a layer's Edit button (switches to Nodes with that sheet loaded).
    • Lens — the lens-corrector tool (§21 / §22): the authoring / calibration / library tool for lens profile files. Reached from the header, or from the Edit lens-correction built-in's + New button (switch to Lens).
  • File relationship is the same for both authoring tools: a sheet file made in Nodes is referenced (copy or link) by a layer in Edit; a lens profile file made in Lens is referenced (copy or link) by the lens-correction panel in Edit. Edit never authors files — it consumes them.
  • Switching sections preserves the current photo and the current selection. Opening a photo in Edit = double-click in Library → switch to Edit with that photo loaded.
  • The no-photo state in §8 (Nodes with blank graph) and in §21.6 (Lens with the built-in grid) is resolved: the user picks a photo in Library first. Edit with no photo selected shows an empty-state prompt to go to Library. Nodes and Lens can still be opened with no photo for pure authoring (blank graph / test grid).
  • The bottom filmstrip is shared across all four sections and shows the current source/selection; clicking a filmstrip frame loads that photo into the active section. This fills the filmstrip slot referenced in Hole #6.

14.4 Import flow

  • The Import button (top toolbar) opens the import dialog.
  • Dialog fields:
    • Source folder picker (a single root; a recursive checkbox is on by default).
    • Candidate list — every supported image file under the root, each with an include checkbox (all on by default) and a small size/date preview.
    • Duplicate handling — file path is the unique key. Paths already in the catalog are shown greyed with an "already imported" badge and excluded by default. An optional "detect duplicates by content" checkbox (off by default, for speed) marks same-content-different-path files with a "duplicate" badge and excludes them by default.
    • Read edits from XMP sidecar checkbox (off by default) — if a .xmp-style sidecar exists beside a file, seed that photo's edit list from it instead of starting empty.
  • Preview generation — one thumbnail and one standard preview are built per file at import.
  • Progress — a progress bar with per-file status (EXIF scan → thumbnail → preview), a running count, and a Cancel button (cancel stops remaining files; already-imported files stay in the catalog).
  • Supported formats — common still formats (JPEG, PNG, TIFF) plus camera RAW via the platform decoder. The exact format list and RAW coverage are implementation/stack and remain deferred (§10).
  • On completion the new photos appear in All Photos and under their folder in the source tree; the most-recently imported photo is selected.

14.5 Browse & select

  • Single select: click a thumbnail → loads it into the metadata panel; double-click → switches to Edit with that photo loaded.
  • Multi-select: Ctrl/Cmd-click toggles individual photos; Shift-click extends a range. The selection persists across module switches and is the foundation for the later batch operations (§14.8 / Hole #8).
  • Keyboard: arrow keys move focus/selection in the grid; Enter opens in Edit; Esc clears the selection.
  • Search: the toolbar search box filters by filename (substring, case-insensitive) within the current source. Advanced filters (date/camera/lens/etc.) are deferred (§14.8).
  • Sort: Capture date (default) / Import date / File name, with ascending/descending.

14.6 Missing files & relink

Because files are referenced in place, a file moved/renamed/deleted outside the app becomes missing:

  • Missing photos show a ? badge on the thumbnail and are greyed in the grid; the metadata panel shows "File not found at <path>."
  • Per-photo relink: a "Find missing file" command opens a file picker; choosing the new location updates the catalog path for that photo.
  • Folder relink: a "Find missing folder" command on a missing folder in the source tree re-roots all its descendant paths in one action (the common case — a whole shoot moved).
  • Missing photos keep their edit list and previews; they are not deleted from the catalog automatically.

14.7 Relationship to the edit list & sheets

  • The catalog is where each photo's edit list (global + per-mask, §2/§3) is persisted. The Edit module reads and writes the edit list through the catalog; nothing about the edit list itself changes from §3–§7.
  • Sheet files stay separate — they live in the user's sheets folder (§2) and are not part of the catalog. The catalog stores, for each layer in each photo's edit list: the sheet file path, its phase (§6), and whether it is a copy or link (plus the copied snapshot, if copy).
  • A linked layer whose sheet file is missing/moved behaves the same way as a missing photo: the layer header shows a broken-link badge and offers relink (parallels §14.6). The full linked-sheet broken-link and versioning UX stays in the holes list (#11, #12).

  • Each effect node exposes a single interface: render(inputImage, params) → outputImage.

Chromium's WebGPU gives compute-shader support and direct access to the GPU without leaving the existing process model. If a future version needs features WebGPU cannot express (e.g. vendor-specific extensions or off-process compute), the alternative is a small native addon (Rust + wgpu, or NAPI) that exposes the same render(inputImage, params) → outputImage interface. 16. Lens Corrector — core concept & terminology

The Lens Corrector is the subsystem behind the Lens header module and the Lens Correction built-in edit in the Edit module. Per-lens profile files are tagged by (focal length, aperture), with linear interpolation between profile points. The lens-correction math (distortion, vignetting, CA, colour vignette) runs on the same CPU stack as the rest of the app (§15).

Raw editors typically apply one static lens-correction profile per lens. But a lens's distortion and vignetting change with focal length and aperture, so a single profile over-corrects one image and under-corrects another shot on the same lens. The Lens Corrector fixes this by storing multiple profile points per lens — each tagged with a (focal length, aperture) pair — and linearly interpolating between the nearest points for every photo. The more points the user adds, the more accurate the result becomes. The system auto-detects the lens and shooting parameters from EXIF and applies the interpolated correction as the first step in the pipeline (§6.1). Profiles are shareable as JSON packs so communities can distribute calibration data.

Design intent: treat lens correction not as a single dial but as a surface across focal length × aperture. The user's job is to populate that surface with points; the app's job is to read the surface smoothly. Calibration is a first-class workflow, not a one-time setup.

16.1 Why it matters

  • Real lenses are not uniform. A 24–70mm f/2.8 zoom has dramatically different barrel distortion at 24mm vs 70mm, and vignetting at f/2.8 is much heavier than at f/8. A single-profile correction must average — and therefore mis-corrects most frames.
  • Calibration improves with use. Unlike most edits, a lens profile gets better the more the user shoots with that lens. Every photo at a new (FL, aperture) is an opportunity to fill in the surface.
  • Community scales calibration. Once one user calibrates a popular lens, sharing the pack saves every other owner hours of work.

16.2 Terminology

  • Lens entry — one physical lens in the user's library. Identified by make + model (and optional serial / variant notes).
  • Lens profile file — a file in the user's lens folder holding one lens entry. Authored in the Lens module (§21 / §22). Referenced (copy or link) by the Edit lens-correction panel (§18), the same way a node-sheet file is referenced by a layer in Edit. A profile pack (below) is the multi-lens export / import container.
  • Profile point — a single calibration sample at a specific (focal length, aperture) pair, holding correction values for all four parameter types.
  • Profile surface — the implicit 2D function over (focal length, aperture) defined by the lens's profile points and the interpolation rule. What gets applied to a photo.
  • Calibration — the act of creating or refining a profile point by adjusting sliders against a reference image.
  • Profile pack — a JSON file containing one or more lens entries for export / import.
  • Active lens — the lens entry the corrector is currently using for the loaded photo (auto-detected or manually chosen).
  • Coverage — a visual indicator of how well the profile surface is sampled across the (FL, aperture) range the lens actually supports.

18.6 Copy / link semantics

  • The lens-correction panel references a lens profile file from the user's lens folder. On adding / swapping a file, the app asks copy or link? — the same prompt as a custom sheet layer (§3).

19. EXIF auto-detect & interpolation

19.1 EXIF fields read

EXIF tagUsed for
Make + Model (camera)Sanity check / display.
LensMake + LensModelMatch a lens entry in the library.
FocalLengthTag the active point on the (FL, aperture) surface.
FNumberTag the active point on the (FL, aperture) surface.
LensSerialNumber (if present)Disambiguate multiple copies of the same lens model.

19.2 Match algorithm

  1. Read LensMake + LensModel from the photo.
  2. Find lens entries where make + model match (case-insensitive, trimmed).
  3. If multiple matches, prefer the one with a matching LensSerialNumber; otherwise prompt the user to pick (Hole #25).
  4. Read the photo's FocalLength and FNumber → these define the (FL, aperture) coordinate on the lens's profile surface.
  5. Compute the interpolated correction (see §19.4).
  6. Apply it as the first pipeline stage (§6.1).

19.3 Missing EXIF

If the photo lacks LensModel or FocalLength (scanned images, heavily cropped JPEGs), the corrector falls back to manual mode and shows a notice: "EXIF incomplete — choose lens and (FL, aperture) manually." See Hole #26.

19.4 Interpolation rule

When a photo is at a (FL, aperture) coordinate that doesn't exactly match a profile point, the corrector finds the nearest profile points in (FL, aperture) space and linearly interpolates their parameter values. Linear is the chosen model: simple, predictable, and behaves well with sparse data — a user with one or two points gets sensible corrections, and the surface refines gracefully as more points are added.

19.5 How neighbours are found

  1. Compute the photo's (FL, aperture) coordinate.
  2. For each profile point, compute a distance. Default metric: weighted Euclidean with weights that emphasise focal length — empirically, distortion varies more sharply with zoom than with aperture. 
      d = sqrt( ( (FL - FL_p) / FL_range )² + ( (A - A_p) / A_range )² )
  3. Take the two nearest profile points (or one if only one exists; or zero if none exist).
  4. Weight each by inverse distance (with epsilon against division-by-zero at exact match).
  5. Linearly combine the per-parameter values: distortion, vignetting, CA, colour vignette — each interpolated independently (no cross-parameter coupling).

19.6 Out-of-range behaviour

If the photo's (FL, aperture) is outside the convex hull of the lens's profile points, the corrector uses the single nearest profile point with a "Far from calibration" badge on the panel (no extrapolation). See Hole #27 for the visual treatment.

19.7 Coverage indicator

The coverage heatmap lives in the Lens module (calibration screen §21 / library manager §22), not in the compact Edit panel. It shows density of profile points across (FL, aperture) for the active lens:

  • Empty cells = no nearby points. Photos at those coordinates will use the nearest point only.
  • Dense cells = many points; interpolation is most accurate here.
  • Current photo's cell is highlighted so the user can see at a glance how well-supported the current shot is.

The indicator's job is to motivate the user to fill gaps in the surface.

20. Per-photo overrides

Even with a well-calibrated surface, a particular photo may need a tweak (a focus shift, an off-calibration sample). The two basic sliders in the Edit panel (§18.4 — Distort, Vignette) act as per-photo overrides on top of the referenced profile's interpolated correction. The full per-parameter surface (k1/k2/k3, falloff curve, CA, colour vignette) is edited in the Lens module on the profile file, not per-photo in Edit.

20.1 How overrides combine with interpolation

For each parameter, the corrector stores a per-photo override value (default = identity / "use interpolation"). The final value applied is:

final = interpolated(profile surface) ⊕ per-photo override

where ⊕ is parameter-appropriate combination:

  • Distort slider — scales the profile's overall distortion correction for this photo (additive offset on the combined k1/k2/k3/scale result).
  • Vignette slider — scales the profile's vignetting correction for this photo (multiplicative; a 0.1 nudge = 10% more / less correction).

CA and colour vignette are not per-photo adjustable in v1 — they follow the referenced profile's interpolated values exactly. Exposing them per-photo is deferred (§10).

20.2 Override affordances

  • When a slider is at its default (no override), it shows in the "interpolated" state — no badge.
  • When the user moves it, a small badge appears next to the slider label: "overridden".
  • Double-clicking the label resets the override to "use interpolation" (back to interpolated state).
  • A "Reset all overrides" link at the bottom of the panel clears every per-photo adjustment in one click.

20.3 Storing overrides

Per-photo overrides live on the photo's edit metadata, not in the lens entry. Editing the lens profile does not overwrite a photo's overrides, and removing a photo's overrides does not affect the lens profile. They are independent layers of state.

Lens data can be exported as JSON packs and imported from packs. This is how communities share calibration data.

23.1 Export

  • Per-lens export — one lens, all its points. File: {lens-slug}.lenspack.json.
  • Whole-library export — all lenses, all points. File: all-lenses.lenspack.json.
  • Selective multi-lens export — checkboxes in the library manager; file: {n}-lenses.lenspack.json.

23.2 Import

  • File picker accepts one or more .lenspack.json files.
  • For each lens in the pack, the app compares make + model + variant against the local library:
    • No match → import as a new lens entry.
    • Exact match (same make + model + variant) → merge profile points: points whose (FL, aperture) is new are added; existing points are left untouched unless the user picks "overwrite" or "merge newer-wins" (see Hole #29).
  • Author and notes from the pack are preserved as metadata on the lens entry; users can see "sourced from hugh's pack, 2026-06-10".

23.3 Pack provenance

Every pack carries an author field and per-point calibratedFrom + notes. The lens library surfaces this in the manager (§22) so users know where a profile came from. The point of a community is knowing which calibrations to trust.

{
  "format": "lens-profile-pack",
  "version": 1,
  "author": "hugh",
  "exportedAt": "2026-06-25T14:30:00Z",
  "notes": "Calibrated on DNG test charts, 2m subject distance.",
  "lenses": [ /* LensEntry, see §23.5 */ ]
}

23.7 Versioning

Format version is bumped on backward-incompatible changes only. The app refuses to import a pack whose version is higher than it understands, and warns (but accepts) on packs whose version is lower (best-effort load, missing fields use defaults). See Hole #30.

Holes & missing info

Gaps, contradictions, and unspecified details found while compiling this spec. Severity: high blocks v1; med needs an answer before build; low can wait.

#1 — Effect-node set for the palette high

Built-in sheets are defined (Basic, Colour, HSL, Tone Curve, Detail), but the individual effect nodes available in the node palette to build custom sheets are not. What nodes ship in v1? (Exposure, Curves, Channel Mixer, Blend, HSL, Split Tone, Sharpen, Noise, Vignette, …?) The §13.3 palette lists categories but no contents. Without this, users can't build anything.

#2 — Free-param editing UX high

When an effect-node param is free (not bound to a UI vertex), where/how is its value set inside the node editor? Options: inline field on the node body, a separate node-properties panel, double-click the node to open a param sheet. Not specified — and it's the most common authoring action.

#3 — UI vertex authoring details high

§4/§11.3 say vertices support "custom label, custom range, reorder" but don't say where those are edited. Is there a vertex-properties panel? Inline edit on the UI node? How do you set a Dropdown/enum's option list? How do you define a Curve vertex's default curve? This is core to the killer feature.

#4 — Mask panel UI & creation med resolved

Resolved by §7.1–§7.3. The mask panel lives as a global | mask | crop tab switcher at the top of the Edit module's right panel. The mask tab contains a ＋ New mask shape picker, a mask list with enable/name/type/actions, and the selected mask's edit list. Mask shape editing is triggered by an "Edit shape" action per mask, which activates the mask-drawing controls on the photo canvas.

#5 — Mask list management med resolved

Resolved by §7.2. The mask list supports multiple masks per photo, default names with inline rename, drag-to-reorder, duplicate, delete, and enable/disable toggles. Inverting and feathering are handled per mask shape in the shape-editing UI.

#6 — Main editor left/centre not specified med resolved

Resolved by §3: the Edit module has no left panel. The centre is a single photo canvas with a top toolbar (zoom dropdown + full-screen button). The right panel places the histogram at the very top, above the global/mask/crop tabs. The shared filmstrip and 4-way module switcher are defined in §14.2/§14.3.

#7 — Photo import / library / catalog med resolved

Resolved by §14. The Library module is now specified: a SQLite catalog (source of truth for photos, paths, metadata, edit lists, preview index), photos referenced in place (never copied), a 4-way Library | Edit | Nodes | Lens module switcher, a folder-mirroring source list + thumbnail grid + read-only metadata panel, a full import dialog (source folder, recursive, duplicate handling by path with optional content-hash, optional XMP read, progress + cancel), browse/select with multi-select, and missing-file/folder relink. Scope of this pass was import + browse + select; organising (albums, keywords, ratings) and batch operations are recorded as deferred in §14.8.

#8 — Multi-photo & batch operations med deferred

Can you select multiple photos and apply a sheet to all? Copy/paste edits between photos (Lightroom's "previous edit" / sync)? Paste a single sheet layer vs the whole stack? Not specified. Note: the multi-select model these operations need is now defined in §14.5; the batch operations themselves remain deferred (§14.8) and will be specced in a later pass.

#9 — Node-editor wiring interactions med

Deferred earlier. Pan/zoom gestures, drag-to-connect from port to port, delete an edge, reconnect (drag endpoint vs redraw), multi-select nodes, copy/paste nodes, delete nodes, align/snap, keyboard modifiers. All unspecified.

#10 — Pin × copy/link interaction med

Pin = "global panel slot, stays visible across all photos; slider values stored per photo." But if a pinned sheet was added as a copy (per-photo), does the pinned slot show that one photo's copy, or the sheet file's defaults? And for a linked pinned sheet, do all photos share one slider-value set or per-photo values? The "stored per photo" clause conflicts with "linked = live to the file." Needs a clear rule.

#11 — Linked-sheet deletion / broken links med

If a user deletes (or moves/renames) a sheet file from their folder, what happens to photos that link to it? Broken-link UI in the layer header? Auto-convert to copy? Greyed-out panel? Not specified.

#12 — Linked-sheet versioning / migration med

Editing a linked sheet updates all linked photos. What if v1 had a "Warmth" slider and v2 removes it — do photos that had a Warmth value break? Is there a migration path, a "frozen at last-compatible version" fallback, or just graceful degradation? Not specified.

#13 — Built-in sheet duplication low

Built-ins are locked (not editable), but can a user duplicate a built-in to start a custom sheet from it ("Duplicate Basic → tweak")? Natural workflow; currently unaddressed. The header ops in §8 (Save As / Duplicate) apply to the open sheet — does "Duplicate a locked built-in" produce an unlocked copy?

#14 — Curve / Color / Dropdown control rendering low

A Curve vertex shows in the main panel as an editable curve widget — points? Bezier? RGB vs luminance? A Color vertex — swatch + picker (RGB/HSV/hex/native OS)? A Dropdown — how are its options defined in the node editor? All unspecified but low-risk to fill in later.

#15 — Free-param visibility in main editor low

If a param is free, is its value shown anywhere in the main editor, or only inside the node editor? Probably the latter (only bound controls surface), but worth confirming so users aren't confused why a sheet "does something" they can't see a slider for.

#16 — Sheet library manager low

§13.2 (header ops) and §8 cover the sheet's own header operations (Save, Save As, Rename, Duplicate, Delete) applied to the currently open sheet. But there's no way to manage sheets you're not currently editing (rename/delete/duplicate/categorise/organise) except by opening each one. A small library manager is implied but unspecified — the lens library manager (§22) is the analogue to copy.

#17 — Reorder-handle scope low

§5 says "Reorder handle — drag within the Custom Edits section." §6 says customs always display at the bottom. Confirm: the handle cannot drag a custom above a built-in in the panel (only reorders within the custom section), and phase is what decides actual pipeline position. Worth a sentence to prevent confusion.

#18 — Nested sheets / sheet-as-node low

Can a sheet reference another sheet as a node (nested sheets / sub-graphs)? §6 chose "stacked layers, not master graph with sub-graphs," which implies no — but it's a natural power-user feature. Confirm "no nested sheets in v1" explicitly.

#19 — Sheet name collisions low

Two sheets with the same display name in the folder — how does the dropdown disambiguate (show filename? author? timestamp)? What happens on duplicate save?

#20 — Error states low

When a node fails (bad param values, unsupported bit depth, evaluation error), what does the user see? Red node? Warning badge? Fallback to last good output? Not specified.

#21 — Keyboard shortcuts low

None defined. At minimum: node-editor toggle, fit-view, delete selection, undo/redo, add-node quick-search. Can wait.

#22 — Slider tooltips & value display low

Hover a slider in the main editor — show current value? Range? Unit? Lightroom shows value on hover + while dragging. Confirm we match.

#23 — Phase naming vs built-in contents low

Phase bucket is "After exposure correction" but the built-in there is "Basic" which does Exposure and Contrast/Highlights/Shadows/Whites/Blacks. Minor naming mismatch — "After Basic" might read clearer than "After exposure correction." Cosmetic but worth aligning.

Lens Corrector holes

#24 — Unknown-lens state in the panel high resolved

Resolved by §18.3. When Auto from EXIF is on and no lens profile file matches, the dropdown shows empty / "Unknown lens — create one?"; the user can pick an existing file manually or hit + New to switch to the Lens module and create a matching profile. The three original candidates (empty-state CTA, pre-filled dropdown, generic banner) collapse to the chosen one.

#25 — Multiple matches on LensMake + LensModel high resolved

Resolved by §19.2 step 3. If multiple lens entries match LensMake + LensModel and no LensSerialNumber disambiguates, the app prompts the user to pick. Remaining detail (whether and how the choice is remembered per-photo) is implementation polish, not a blocking gap.

#26 — Missing EXIF fallback high partial

Partly closed by §19.3: the corrector falls back to manual mode and shows a "EXIF incomplete — choose lens and (FL, aperture) manually" notice. Remaining unspecified: whether Auto is explicitly disabled (vs silently ignored), the exact shape of the two (FL, aperture) numeric inputs in the compact Edit panel (§18.1 currently shows only Distort + Vignette sliders), and whether the correction then uses the closest profile point exactly or still interpolates from the manual coordinate. (Cross-referenced from §19.3 / §18.1.)

#27 — Out-of-range visual treatment med

When the photo's (FL, aperture) is outside the convex hull of profile points (§19.6), the user should see a "Far from calibration" badge on the panel. Confirm the badge's colour, position, and tooltip ("Nearest point is at 35mm f/2.8; you are at 200mm f/5.6 — consider adding a profile point here").

#28 — Profile-point raw-value inspect med partial

Partly closed by §22.3. The per-point "Inspect raw values" action opens a popover showing the JSON for that point. Remaining unspecified: whether the popover is explicitly read-only (no inline edit), and whether it has a "Copy to clipboard" button (useful for debugging and for power users who hand-edit packs).

#29 — Pack-import conflict policy med

Conflicts on overlapping (FL, aperture) points within an "exact match" still need a single, settled rule. §23.2 currently offers "overwrite" or "merge newer-wins" as user choices; the earlier draft here proposed a per-point prompt ("keep local" / "use imported" / "skip", default skip to preserve local work). These are not the same UX. Pick one before build and align §23.2 with this entry. (Cross-referenced from §23.2.)

#30 — Pack format versioning policy med resolved

Resolved by §23.7. Format version is bumped only on backward-incompatible changes. The app refuses to import a pack whose version is higher than it understands (clear error), and warns but accepts on a lower version (best-effort load, missing fields use defaults). No automatic migration in v1.

#31 — Pipeline interaction with "First / at start" customs med resolved

Resolved by §6.1. Lens correction is stage 0, ahead of all customs including phase=first (stage 1). A custom sheet with phase=first still runs after lens correction. Phase choice cannot skip lens correction — only the corrector panel's Enable toggle (§18.2) can bypass it.

#32 — Coverage heatmap resolution med

§18.1 / §19.7 describe the coverage heatmap, which lives in the Lens module (not in the compact Edit panel). Confirm: the cells are binned (e.g. one cell per (f-stop × 5mm FL bin)), not continuous. The "current photo" cell is highlighted. Hovering a cell shows the count of profile points in that bin.

#33 — Vignetting curve interpolation across points med

The vignetting curve (§17.4) is a control curve — it has its own point list. When interpolating between two profile points at different (FL, aperture), the curves have potentially different point counts. Confirm: interpolation samples each curve at a fixed set of x-values (e.g. 0, 0.1, 0.2, …, 1.0), linearly interpolates the y-values, and reconstructs a curve from the combined samples.

#34 — Per-photo override storage schema low

§20.3 says per-photo overrides live on photo edit metadata, not in the lens entry. Confirm the storage shape: a flat key/value object keyed by "<lens-id>.<parameter>", e.g. "9b1c…vignetting.strength": 0.05. Resolves cleanly with the photo's reference to the active lens at apply time.

#35 — Calibration undo low

§9 defines per-sheet undo. Lens correction is not a sheet — confirm: per-photo slider changes in the panel follow the main editor's undo stack (i.e. undo/redo in the main editor undoes lens-corrector slider drags on the active photo). Calibration-screen edits (adding/editing a profile point) get their own undo inside the calibration screen, and committing the point is a single undoable step in the library.

#36 — Disabled Auto-detect behaviour low

When the user turns off "Auto from EXIF", the panel should expose manual (FL, aperture) inputs. Confirm: those inputs persist as per-photo overrides (§20), so the user can deliberately assign a non-EXIF (FL, aperture) to a photo (e.g. scanned film where the photographer knows the lens used).

#37 — Calibration against a non-current photo low

§21.3 option 1 uses "the loaded photo". Confirm: calibration can also be run against any other photo the user picks (e.g. "I have a dedicated test-chart shot from yesterday — let me calibrate against that"). A simple file picker in the calibration screen suffices; the loaded photo remains unchanged.

#38 — Pack-author display low

§23.3 says the library surfaces the pack's author. Confirm the display: in the lens library manager (§22), the lens entry shows a small attribution line ("profiles sourced from hugh, 2026-06-10") — not a modal, not a banner, just a quiet line under the lens name.

#39 — Per-parameter enable toggles within the corrector low

Should each of the four correction groups (Distortion / Vignetting / CA / Colour Vignette) have its own enable toggle, in addition to the Edit panel's global On/Off? The Edit panel (§18) no longer exposes the four groups — only Distort + Vignette sliders — so per-group enables would live in the Lens module (on the profile / calibration screen, §21), letting the user bake "distortion on, CA off" into a profile. Confirm location and whether a per-group on/off is stored on the profile file vs applied per-photo.
